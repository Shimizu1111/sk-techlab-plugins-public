# security-check

インターネット公開前のセキュリティチェックを実施する。
外部から不正な操作をされないよう、精度高く網羅的に検査する。時間をかけてでも徹底的にチェックする。

## 起動条件

ユーザーが `/security-check` を実行したとき

## 入力

- 対象プロジェクトのパス（省略時はカレントディレクトリ）
- 対象の技術スタック（不明な場合は自動検出する）

## 実行手順

### 1. プロジェクト構成の把握

- 使用言語・フレームワーク・ランタイムを特定する
- エントリポイント、ルーティング、API定義を把握する
- デプロイ構成（Dockerfile、CI/CD等）を確認する

### 2. 秘密情報の漏洩チェック

- APIキー、トークン、パスワード、秘密鍵がコードにハードコードされていないか
- `.env` ファイルが `.gitignore` に含まれているか
- git履歴に秘密情報がコミットされていないか（直近のコミットを確認）
- 公開リポジトリにすべきでないファイルが含まれていないか

### 3. 認証・認可

- 認証が必要なエンドポイントに認証チェックがあるか
- セッション管理は安全か（トークンの有効期限、HTTPOnly、Secure属性）
- RBAC/権限チェックが適切に実装されているか
- デフォルトの管理者アカウント・パスワードが残っていないか

### 4. インジェクション攻撃

- SQLインジェクション: プリペアドステートメント/パラメータ化クエリを使っているか
- XSS: ユーザー入力のサニタイズ・エスケープがされているか
- コマンドインジェクション: `exec`/`eval`/`system` 等でユーザー入力を直接使っていないか
- パストラバーサル: ファイルパスにユーザー入力を直接使っていないか
- テンプレートインジェクション（SSTI）: テンプレートエンジンの安全な使用

### 5. API・通信のセキュリティ

- CORS設定が適切か（ワイルドカード `*` になっていないか）
- CSRFトークンが実装されているか
- レートリミットが設定されているか
- HTTPSが強制されているか
- セキュリティヘッダー（CSP, X-Frame-Options, HSTS等）が設定されているか

### 6. データ保護

- パスワードが適切にハッシュ化されているか（bcrypt/argon2等）
- 個人情報・機密データが平文で保存されていないか
- ログに機密情報が出力されていないか
- エラーメッセージで内部情報が漏洩していないか（スタックトレース等）

### 7. 依存パッケージ

- `npm audit` / `pip audit` / `bundle audit` 等で既知の脆弱性をチェック
- 古いバージョンの依存パッケージがないか
- 不要な依存が含まれていないか

### 8. インフラ・構成

- Dockerfileのベストプラクティス（rootユーザーで実行していないか等）
- ファイアウォール・ネットワーク設定の確認
- 不要なポートが公開されていないか
- デバッグモードが無効化されているか
- 環境変数のバリデーション

### 9. ビジネスロジック

- 権限昇格が可能なフローがないか
- 決済・課金周りの不正操作（金額改ざん、数量改ざん等）
- レースコンディションによる二重処理

## 出力フォーマット

チェック完了後、以下の形式でレポートを出力する:

```markdown
# セキュリティチェックレポート

## 対象
- プロジェクト: {プロジェクト名}
- 技術スタック: {検出されたスタック}
- チェック日: {日付}

## サマリー
- 危険（Critical）: {件数}
- 警告（Warning）: {件数}
- 情報（Info）: {件数}
- 問題なし: {件数}

## 検出項目

### [Critical] {タイトル}
- 場所: {ファイル:行番号}
- 説明: {何が問題か}
- リスク: {攻撃シナリオ}
- 対策: {具体的な修正方法}

### [Warning] {タイトル}
...

### [Info] {タイトル}
...

## チェック済み項目（問題なし）
- {チェック項目名}: OK
...

## 推奨アクション（優先度順）
1. {最優先で対応すべきこと}
2. ...
```

## 注意事項

- 検出した問題は必ず具体的なファイルパスと行番号を示す
- 修正方法は具体的なコード例を含める
- 該当しないチェック項目（例: DBを使っていないプロジェクトのSQLi）はスキップし、その旨を記載する
- チェック漏れより過検出を優先する（疑わしいものは報告する）
- 自動修正は行わない。レポートのみ出力し、修正はユーザーの判断に委ねる
