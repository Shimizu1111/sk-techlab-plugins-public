# Claude Code 30日マスター

このスキルは30日間のClaude Code学習カリキュラムです。

## 使い方

- `/learn-claude-code` → 次の未完了レッスンを表示して解説する
- `/learn-claude-code status` → 全体の進捗を表示する
- `/learn-claude-code done` → 現在のレッスンを完了にする
- `/learn-claude-code day 15` → 指定した日のレッスンを表示する
- `/learn-claude-code reset` → 進捗をリセットする

## 動作

1. `.claude/skills/learn-claude-code/progress.json` を読み込む（なければ初期化する）
2. 引数に応じて処理を分岐する：
   - 引数なし: 次の未完了Dayのレッスンを表示・解説する
   - `status`: チェックリスト全体を表示し、完了/未完了を一覧する
   - `done`: 現在取り組み中のDayを完了にしてprogress.jsonを更新する
   - `day N`: 指定Dayのレッスンを表示する
   - `reset`: progress.jsonを初期状態に戻す

## レッスン表示時のルール

- 非ITの人にもわかるように**たとえ話や日常の例え**を使って説明する
- 「なぜこれが重要か」を最初に伝える
- 実際に試せる**ハンズオン課題**を1つ含める
- 課題を完了したら `/learn-claude-code done` で完了にするよう案内する

## カリキュラム（30日分）

進捗管理は `progress.json` で行う。以下がカリキュラム内容：

### Week 1: 基礎と考え方（まず「なぜ」を理解する）

- **Day 1: コンテキストウィンドウとは何か**
  Claudeの「作業机」の広さ。ファイルを読む・会話する・コマンド結果を見る、全てがこの机に乗る。机がいっぱいになると集中力が落ちる。だから管理が最重要。

- **Day 2: 良いプロンプトの書き方**
  「いい感じにして」ではなく「何を・どこの・どうやって確認するか」を伝える。レストランで「美味しいもの」と注文するか「辛さ控えめの麻婆豆腐」と注文するかの違い。

- **Day 3: 検証手段を与える**
  Claudeに「答え合わせの方法」を渡す。テスト実行、lint通過、スクリーンショット比較など。自分で確認できるClaudeは精度が段違い。

- **Day 4: Plan Mode（計画モード）**
  いきなり工事を始めず、設計図を描いてから着手する。Ctrl+Gで切り替え。探索→計画→実装→確認の4ステップ。

- **Day 5: /compact と /clear**
  `/compact`は机の整理整頓（要約して空きを作る）。`/clear`は机を完全に片付けて新しい作業を始める。タスクが変わったら`/clear`。

- **Day 6: 軌道修正の方法**
  Escで停止、Esc×2で巻き戻し。2回修正してもダメなら、やり直した方が早い。「迷ったら戻る」を徹底する。

- **Day 7: CLAUDE.mdの書き方**
  プロジェクトのルールブック。毎回Claudeが読む「申し送りメモ」。短く・具体的に・Claudeが間違えることだけ書く。

### Week 2: 環境を整える（道具を揃える）

- **Day 8: Permission設定（権限管理）**
  Claudeに「何を許可するか」を決める。毎回「実行していい？」と聞かれるのを減らす仕組み。auto mode / allowlist / sandbox の3段階。

- **Day 9: Hooks（自動実行フック）**
  「ファイルを保存したら自動でフォーマット」のような自動処理。CLAUDE.mdのお願いと違い、100%確実に実行される。

- **Day 10: Skills（スキル定義）**
  よく使う手順をスラッシュコマンドにする。このカリキュラム自体がSkillの実例。チームで共有できる。

- **Day 11: MCP連携（外部ツール接続）**
  Claude CodeをNotion、Figma、DB等の外部サービスと繋ぐ。「コンセントを挿す」感覚で機能を追加できる。

- **Day 12: CLIツールの活用**
  gh（GitHub）、aws、gcloud等をClaude経由で使う。Claudeに`--help`を読ませれば未知のツールも使いこなせる。

- **Day 13: @参照とリッチ入力**
  `@ファイル名`で直接参照、画像の貼り付け、URLの提示、パイプで結果を流し込む。Claudeに渡す情報を豊かにする方法。

- **Day 14: Settings全体像**
  settings.jsonの構造を理解する。グローバル（全プロジェクト共通）・プロジェクト（チーム共有）・ローカル（自分だけ）の3層。

### Week 3: 並列化とスケール（1人を10人にする）

- **Day 15: Subagent（サブエージェント）**
  「部下に調査を任せる」感覚。別のコンテキストで動くので、メインの作業机が散らからない。最重要テクニック。

- **Day 16: カスタムSubagent定義**
  `.claude/agents/`に専門家を定義する。セキュリティ担当、テスト担当など。用途特化で精度が上がる。

- **Day 17: Worktree（並行ブランチ作業）**
  1つのプロジェクトで複数の作業を同時に進める。机を2つ用意して、それぞれ別の仕事をするイメージ。

- **Day 18: 非対話モード (-p)**
  `claude -p "命令"`でスクリプトから呼ぶ。CI/CDやcronで自動実行する基盤。対話なしで結果だけ受け取る。

- **Day 19: Fan-outパターン（一括処理）**
  100個のファイルに同じ処理をかける。1つずつ手動でやるのではなく、ループで一気に処理する大量作業の手法。

- **Day 20: Routines（定期実行）**
  Anthropicのクラウド上で定期的にClaudeを走らせる。毎朝のPRレビュー、週次の依存更新など。PCを閉じていても動く。

- **Day 21: セッション管理**
  会話に名前をつけて保存。翌日続きから再開。作業ごとにセッションを分ける整理術。

### Week 4: 実戦パターンと応用（プロの技）

- **Day 22: インタビューパターン**
  「仕様を全部自分で考える」のではなく「Claudeに質問させる」。抜け漏れが激減する設計手法。

- **Day 23: Writer/Reviewerパターン**
  1つ目のセッションで書く、2つ目のセッションでレビュー。自分の書いたコードを自分でレビューしない原則。

- **Day 24: Checkpointing（チェックポイント）**
  Claudeの全操作に自動セーブポイント。ゲームのセーブのように、いつでも任意の時点に戻れる。

- **Day 25: Chrome連携**
  ブラウザを直接操作してUI確認。画面のスクリーンショットを撮って見た目を検証する。

- **Day 26: GitHub Actions / GitLab CI**
  PRが来たら自動でClaudeがレビュー。Issueが立ったら自動でトリアージ。人手ゼロの自動化。

- **Day 27: Channels（外部イベント連携）**
  Discord/Telegram/webhookから実行中のClaudeセッションに情報を送る。外部の出来事をリアルタイムに取り込む。

- **Day 28: Plugins**
  コミュニティが作った拡張機能をインストール。code intelligence等で即座に能力追加。

- **Day 29: コスト管理**
  `/cost`でトークン消費を確認。compact/clear/subagentでコストを抑える戦略。fast modeの使い分け。

- **Day 30: 5つのアンチパターン**
  よくある失敗：(1)話題を混ぜすぎ (2)修正ループ (3)CLAUDE.md肥大 (4)検証なし (5)無限探索。知っていれば避けられる。
