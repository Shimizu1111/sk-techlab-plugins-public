# /threads-tips-research - Claude Code海外ノウハウ・Tipsリサーチ

海外の英語圏で共有されているClaude Codeのノウハウ・Tips・ベストプラクティスを収集し、**指定アカウントのターゲットに刺さる形**で翻訳・蓄積する。threads-draft / x-draft / fb-draft に渡せる投稿ネタとして保存する。

**ニュースではなく「知識」。鮮度より有効性が大事。ただし新しいほど良い。**

## 棲み分け

| スキル | 対象 |
|--------|------|
| /threads-news-research | ニュース（新機能発表・業界動向・速報） |
| /threads-global-research | 海外の活用**事例**（企業がどう使ったか） |
| /threads-tips-research | 海外の**ノウハウ・Tips**（使い方のコツ・ベストプラクティス） |

## 引数

- `$ARGUMENTS` にアカウント名やテーマを渡せる（省略可）

## 実行手順

### Step 0: アカウント選択 & ターゲットセグメント選択

#### 0-1. アカウント選択

`SNS/` 配下のフォルダ一覧を取得し、ユーザーに選んでもらう。
`$ARGUMENTS` にアカウント名が含まれていれば自動で特定してスキップ。

#### 0-2. ターゲットセグメント選択

以下の3段階からユーザーに選んでもらう。`$ARGUMENTS` に含まれていれば自動特定。

| セグメント | 説明 | 関心事 |
|-----------|------|--------|
| **検討段階** | Claude Codeをこれから使い始めようとしている | 導入すべきか、ROI、リスク、他社の成功/失敗、何ができるか |
| **初期段階** | 使い始めたけどいまいち使いきれていない | なぜうまくいかない、すぐできる改善、正しい使い方、期待値調整 |
| **活用段階** | 割と使いこなしている | さらに効果を出す方法、チーム展開、自動化拡大、高度な活用 |

**このセグメント選択がリサーチ全体の方向性を決める。** Step 3のクエリ設計、Step 4のフィルタリングすべてに影響する。

### Step 1: アカウント文脈の読み込み

選択されたアカウントの以下を読み込む:

1. **`アカウント設計.md`** -- ターゲット・ポジショニング・トーン・コンテンツ4本柱
2. **`自社情報.md`**（あれば） -- 自分の事業との接点を見つけるため

**読み込んだ情報から以下を確認し、リサーチの軸にする:**

- **ターゲットは誰か**（例: 非ITの経営者・意思決定者）
- **ターゲットの関心事は何か**（例: 売上・コスト・人材・時間）
- **ターゲットに刺さる言葉は何か**（例: 経営者の言葉、具体的な数字と業務名）
- **ターゲットに刺さらない言葉は何か**（例: 技術用語、エンジニア向けの話）

### Step 2: テーマ確認

ユーザーが指定したテーマがあればそれを使う。なければ「Claude Code全般のノウハウ・Tips」で進める。
迷ったら一言だけ確認する（例: 「プロンプト設計系・ワークフロー系・CLAUDE.md設計系のどれ寄せます？」）。長々と質問しない。

### Step 3: ターゲット視点でクエリを設計し検索

**重要: Step 1で読み込んだターゲット像に基づいてクエリを設計する。**
「Claude Codeの技術Tips」を探すのではなく、「ターゲットの関心事 x Claude Code」の交差点を探す。

WebSearch を **複数クエリ並列** で投げる。**英語ソースを優先**。日本語で既に広く知られているものは除外。

**新しいものを優先するが、鮮度より有効性を重視。**

#### クエリ設計の原則

**Step 0-2で選択したセグメントに合わせてクエリを設計する。**
「Claude Code tips」のような汎用クエリだけでは的外れなコンテンツしか引っかからない。

#### セグメント別クエリ例

**検討段階（これから始める人向け）:**
- `Claude Code business ROI results`
- `Claude Code worth it small business`
- `should I use Claude Code business`
- `Claude Code vs hiring developer cost`
- `Claude Code non-technical getting started`
- `AI coding tool adoption risks business`
- `Claude Code what can it do business owner`
- `Claude Code success story small company`
- `site:reddit.com/r/ClaudeAI worth it business`

**初期段階（使い始めたけど微妙な人向け）:**
- `Claude Code not working as expected`
- `Claude Code beginner mistakes business`
- `Claude Code getting better results tips`
- `why Claude Code output is bad`
- `Claude Code expectations vs reality`
- `Claude Code frustrating how to fix`
- `Claude Code common mistakes non-developer`
- `how to give Claude Code better instructions`
- `site:reddit.com/r/ClaudeAI frustrated tips`

**活用段階（割と使っている人向け）:**
- `Claude Code advanced workflow business`
- `Claude Code scaling team deployment`
- `Claude Code automate more business processes`
- `Claude Code power user business tips`
- `Claude Code replace outsourcing advanced`
- `Claude Code multiple projects management`
- `Claude Code team rollout lessons learned`
- `site:reddit.com/r/ClaudeAI advanced business`

**セグメントに応じたクエリを8割以上にする。** 残り2割は汎用クエリで補完。

優先ソース:
- HackerNews（議論の質が高い）
- Reddit (r/ClaudeAI, r/Anthropic, r/smallbusiness, r/Entrepreneur)
- X / Twitter（英語圏のAI開発者 + ビジネスオーナーの実践知）
- dev.to / Medium / Substack（How-to記事・導入記）
- GitHub（awesome-claude-code系、CLAUDE.md実例、Discussions）
- Anthropic公式ドキュメント・カスタマーストーリー
- YouTube（導入事例・ビジネス向けウォークスルー）

### Step 3.5: 一次ソースの特定（必須）

検索で見つけた記事が**一次ソース（オリジナル）か二次ソース（引用・まとめ）かを必ず判定する。**

**二次ソースの兆候:**
- 「〜によると」「〜が報じた」「〜で話題になっている」等の引用表現がある
- 具体的な事故・事件・研究を紹介しているが、本人の取材ではない
- SEOブログ・テックメディアが別のソースを参照してまとめている

**一次ソースが別にある場合:**
1. 記事内の引用元・リンク先を辿る
2. 必要に応じて追加検索する（人名+事件名、企業名+キーワード等）
3. 一次ソースのURLをTipsファイルの `source_urls` に記録する
4. 二次ソースは `related_urls` や本文中の参考として残してもよいが、メインのソースにしない

**例:**
- XDAの「AIに自由を与えすぎると逆効果」記事 → 二次ソース
- PocketOS創業者Jer Craneの証言、The Registerの速報 → 一次ソース
- **保存すべきは一次ソースのURL**

### Step 4: リサーチルールに基づくフィルタリング

検索結果を **リサーチルール** に基づいて選別する。タイトルだけで判断せず、WebFetch で中身を確認してから判断する。

## リサーチルール（調査の独自ルール）

### 0. ターゲット立場テスト & セグメントマッチ（全ルールに優先・最初に判定）

コンテンツを見つけたら、他の判定の前に2つ問う:

**Q1: 「このコンテンツは"ツールを使う人"向けか、"導入を判断・管理する人"向けか？」**
**Q2: 「Step 0-2で選んだセグメント（検討/初期/活用）の人に刺さるか？」**

#### セグメント別の保存/スキップ基準

| セグメント | 保存対象 | スキップ |
|-----------|----------|----------|
| **検討段階** | 導入判断の材料（ROI、コスト比較、成功/失敗事例、「何ができるか」） | 操作方法、使い方のコツ（まだ使っていない人には不要） |
| **初期段階** | よくある失敗と対策、期待値の調整、すぐ試せる改善法、「こう使うと結果が変わる」 | 高度なワークフロー（まだそこまで行っていない）、導入判断の話（もう使っている） |
| **活用段階** | 高度な活用法、チーム展開、自動化の拡大、効果の最大化 | 基礎的な話（既に知っている）、導入判断の話（もう使っている） |

#### 具体例

**検討段階の人に対して:**
- 「Claude Codeに月$200払う価値はあるか → 外注費月$5,000が不要に」→ **保存**
- 「/clearコマンドで精度が上がる」→ **スキップ**（まだ使ってない）
- 「導入して失敗する会社の3つの共通点」→ **保存**

**初期段階の人に対して:**
- 「AIが思った通りに動かない理由は"指示の出し方"にある」→ **保存**
- 「Claude Codeの市場規模が○○億に」→ **スキップ**（もう使ってる、導入判断は済み）
- 「最初の1ヶ月でやるべきこと3つ」→ **保存**

**活用段階の人に対して:**
- 「チーム5人にClaude Codeを展開した結果」→ **保存**
- 「Claude Codeとは何か」→ **スキップ**（既に知っている）
- 「サブエージェントで複数タスクを並列化して工数半減」→ **保存**

**「翻訳すれば刺さる」は言い訳にしない。** セグメントに合わないコンテンツは、言い換えても行動が変わらない。

### 1. 有効性フィルター

**「これを知ったターゲットが、具体的に次の行動を変えるか？」**

- 「AI導入を決断する」「チームに指示を出す」「予算を確保する」→ 保存対象
- 「へー面白い」で終わる → スキップ
- 技術者しか使えないテクニック → Rule 0でスキップ済みのはず。すり抜けていたらここでスキップ

### 2. ターゲットフィルター

**「このコンテンツを読んだターゲットが、翌日の行動を変えるか？」**

- 「社内でAI導入の話を切り出す」「外注先を見直す」「チームの業務フローを変える」→ 保存対象
- 「すごいね」「面白いね」で終わる → スキップ
- 技術的に面白いだけ → スキップ（Rule 0参照）

### 3. 翻訳の方向性

技術的なTipsでも、ターゲットの関心事に「翻訳」して記録する。

```
x ターゲットの関心事ではない（Rule 0でスキップすべき）:
「CLAUDE.mdにディレクトリ構造とコーディング規約を書くとコンテキストロスが減る」
→ いくら「社内マニュアルと同じ」と翻訳しても、経営者は自分でCLAUDE.mdを書かない。行動が変わらない。

o ターゲットの関心事に合致:
「ある会社が外注していたレポート作成をClaude Codeに切り替えたら、
 月15万円の外注費がゼロになった。
 ポイントは"最初の1件を丁寧にやる"こと。
 1件目の品質が良ければ、残り全部を同じ品質で自動化できる」
→ コスト削減の具体例。経営者が「うちでもやろう」と判断できる。
```

翻訳のポイント:
- **そもそもターゲットの関心事に合致するコンテンツを選ぶ**（翻訳で無理やり刺さらせない）
- **金額・時間・人数の具体的な数字で効果を見せる**
- **「だから経営判断としてこうすべき」まで書く**

### 4. 具体性の水準

| 優先度 | 条件 | 例 |
|--------|------|-----|
| A | 具体的な手順+効果の数字がある | 「この3行をCLAUDE.mdに書くだけでエラー率50%減」 |
| B | 具体的な手順はあるが数字なし | 「サブエージェントを使うときはこう指示すると良い」 |
| C | 概念的だが実践に繋がる | 「コンテキストを絞ることが重要」 |
| D | 抽象的・ポエム | 「AIと協業する心構え」 |

**D（抽象的）は原則スキップ**。

### 5. 「日本ではまだ知られていないか」テスト

- **日本語で検索して上位に出るTipsは除外**（差別化のため）
- 「これ、日本の人はまだ知らないけど…」が使えるネタを優先
- Anthropic公式ドキュメントに書いてあっても、日本で実践している人が少なければ対象

### 6. 再現性チェック

- **誰でもすぐに試せるTipsを優先**（特殊な環境・ツールが必要なものは後回し）
- 「読んだ人が今日から実践できるか？」を判断基準にする
- 複雑なセットアップが必要なものは、簡単に始められる部分だけ抜き出す

### 7. 鮮度（副次的）

- **新しいほど良いが、古くても有効なら対象**
- 直近3ヶ月以内を優先するが、6ヶ月前でも「まだ知られていない有効なTips」なら保存
- Claude Codeの仕様変更で無効になったTipsは除外

### Step 5: Tipsごとに保存

1Tips = 1ファイル。3〜5件を目安。

保存先: `SNS/{アカウント名}/海外Tips/Claude Code/YYYY-MM-DD_<短い英数タイトル>.md`
（ファイル名はOS互換性のため英数字+ハイフン推奨。日本語タイトルはフロントマター/本文に書く）

最後に以下を返答する:
- 収集したTipsの一覧（タイトル + 1行要約）
- 特にSNS向きと思う1〜2本のおすすめ理由
- 次に深掘りすると良さそうなテーマがあれば提案

- 技術→業務への言い換え

## ルール

- **有効性が最重要**。「知って実践すれば変わる」情報だけを対象にする。
- **ターゲットの言葉で要約する**。技術用語はそのまま残さず、業務の言葉に翻訳する。ただしツール名（Claude Code等）はそのまま残す。
- **数字や固有名詞を捏造しない**。確認できないものは「未確認」「ソース上の言及なし」と書く。
- **日本で既に広く知られているTipsは除外**（リサーチルール5参照）。
- **引用は短く**。長文引用は避け、要約 + 短い引用 + ソースURLで構成する。
- **ネタとして弱い場合は無理に保存しない**。「良いTipsが少なかった」と素直に伝える。
- **保存先フォルダが無ければ作成**してから書き込む。
- **リサーチルール1〜7をすべて適用してからファイルに書く**。フィルタリングせずにそのまま保存しない。
- **SNS投稿ネタ候補のフック候補は、アカウントのトーンに合わせる**。threads-draftやx-draftにそのまま渡せるクオリティで書く。
- **再現性を重視**。読んだ人が今日から試せないTipsは価値が低い。
