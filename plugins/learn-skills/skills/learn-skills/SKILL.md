# スラッシュコマンド 30日マスター

Claude Code の**スラッシュコマンド**を1日1つずつ、30日間で全て習得するカリキュラムです。
ビルトインコマンド＋ビルトインスキルの全29個を、初日の俯瞰と合わせて30日で完走します。

## 使い方

- `/learn-skills` → 次の未完了レッスンを表示して解説する
- `/learn-skills status` → 全体の進捗を表示する
- `/learn-skills done` → 現在のレッスンを完了にする
- `/learn-skills day 15` → 指定した日のレッスンを表示する
- `/learn-skills reset` → 進捗をリセットする

## 動作

1. `.claude/skills/learn-skills/progress.json` を読み込む（なければ初期化する）
   - 初期値: `{"current_day": 1, "completed": []}`
2. 引数に応じて処理を分岐する：
   - 引数なし: 次の未完了Dayのレッスンを表示・解説する
   - `status`: チェックリスト全体を表示し、完了/未完了を一覧する
   - `done`: 現在取り組み中のDayを完了にしてprogress.jsonを更新する
   - `day N`: 指定Dayのレッスンを表示する（進捗は変えない）
   - `reset`: progress.jsonを初期状態に戻す（確認してから実行する）

## 引数

$ARGUMENTS

## レッスン表示時のルール

- 非ITの人にもわかるように**たとえ話や日常の例え**を使って説明する
- 「なぜこれが重要か」を最初に伝える
- そのコマンドの**全オプション・使い方のバリエーション**を網羅する
- 実際に試せる**ハンズオン課題**を1つ含める（実際にコマンドを打って体験する）
- 課題を完了したら `/learn-skills done` で完了にするよう案内する

## カリキュラム（30日分）

### Week 1: 全体像とコンテキスト管理

- **Day 1: スラッシュコマンドとは何か（全体俯瞰）**
  Claude Codeの中で `/` から始まるショートカット。全29コマンドの一覧と分類を俯瞰する。「メニュー表」を眺めてから料理を選ぶように、まず全体像を掴む。

- **Day 2: /help**
  困ったときの最初の一手。ヘルプを見る習慣が全ての基礎。「取扱説明書を読む」ことの大切さ。

- **Day 3: /compact**
  Claudeの「作業机」を整理整頓するコマンド。会話が長くなると精度が落ちる→compactで要約して空きを作る。カスタムプロンプトも指定できる。最重要コマンドの1つ。

- **Day 4: /clear**
  作業机を完全リセット。タスクが変わったら迷わずclear。compactとの使い分け：同じ話題を続けるならcompact、話題が変わるならclear。

- **Day 5: /cost**
  「今日いくら使った？」を確認する家計簿。トークン消費量とコストをリアルタイムで把握する。使いすぎ防止の必須コマンド。

- **Day 6: /status**
  現在の設定状態をまとめて確認する「ダッシュボード」。モデル、権限、MCP接続などの状態が一覧できる。困ったときにまず打つと原因が見えることが多い。

- **Day 7: /diff**
  Claudeが変更したファイルの差分を確認する。「何が変わったか」を自分の目で確認する習慣。変更を承認する前に必ず見るべきもの。

### Week 2: 設定とカスタマイズ

- **Day 8: /config**
  テーマ、モデル、その他の設定をサクッと変更する。settings.jsonを直接いじらなくてもUIで設定できる手軽さ。

- **Day 9: /model**
  使用するモデルを切り替える。Opus / Sonnet / Haiku の使い分け。重い作業はOpus、軽い質問はHaikuなど、コストと品質のバランス。

- **Day 10: /fast**
  同じOpusモデルのまま高速出力モードに切り替える。品質は同じで速度が上がる。トグルでON/OFF。急いでいるときに便利。

- **Day 11: /permissions**
  Claudeに「何を許可するか」を管理する。ファイル編集、コマンド実行などの許可レベルを確認・変更する。セキュリティの要。

- **Day 12: /allowed-tools**
  許可するツールを細かく管理する。`Bash(git:*)` のようにパターンで指定できる。permissionsより細かい粒度のアクセス制御。

- **Day 13: /hooks**
  「ファイル保存時に自動フォーマット」のような自動処理を管理する。CLAUDE.mdの「お願い」と違い、100%確実に実行される仕組み。

- **Day 14: /vim**
  vimキーバインドモードの切り替え。vim使いには必須。通常モードとの切り替えがトグルでできる。

### Week 3: プロジェクトと外部連携

- **Day 15: /init**
  CLAUDE.mdを自動生成するプロジェクト初期化コマンド。プロジェクトのルールブックの雛形を作る。新しいプロジェクトで最初に打つコマンド。

- **Day 16: /memory**
  Claude Codeのメモリ（自動記憶）を管理する。会話を跨いで覚えておくべきことの確認・編集。MEMORY.mdとの関係。

- **Day 17: /review**
  コードレビューをClaudeに依頼する。変更差分を分析して問題点・改善案を指摘してくれる。自分のコードの第二の目。

- **Day 18: /mcp**
  MCP（Model Context Protocol）サーバーの管理。Notion、GitHub、DB等の外部サービスとの接続を追加・削除・確認する。

- **Day 19: /pr-comments**
  PRのコメントを取得して確認する。GitHub上のレビューコメントをClaude Code内で直接読み、対応できる。

- **Day 20: /doctor**
  Claude Codeの健全性チェック。アップデーターの状態確認、接続問題の診断など。「調子が悪いな」と思ったらまずdoctor。

- **Day 21: /terminal-setup**
  ターミナル統合のセットアップ。Claude Codeとターミナルの連携を最適化する。シェル補完やプロンプト統合の設定。

### Week 4: 認証・ビルトインスキル・総仕上げ

- **Day 22: /login**
  Claude Codeの認証を行う。APIキー、OAuth、チームアカウントなど、認証方式の種類と使い分け。

- **Day 23: /logout**
  ログアウトしてセッションをクリアする。アカウント切り替え時や共有マシンでの安全な終了方法。

- **Day 24: /bug**
  バグを報告する。Claude Codeの問題を開発チームにフィードバックする方法。再現手順の書き方のコツ。

- **Day 25: /simplify**
  変更したコードの品質・効率をレビューして改善するビルトインスキル。リファクタリングの相棒。コード変更後に打つ習慣をつける。

- **Day 26: /update-config**
  settings.jsonを対話的に設定変更するビルトインスキル。hooks、permissions、環境変数など、/configより高度な設定変更に使う。

- **Day 27: /claude-api**
  Claude APIやAnthropic SDKを使ったアプリ開発を支援するビルトインスキル。APIを使うコードを書くときに呼ぶと精度が上がる。

- **Day 28: /loop**
  プロンプトやコマンドを一定間隔で繰り返し実行するビルトインスキル。定期的なステータス確認、監視タスクなどに使う。

- **Day 29: /schedule**
  cronスケジュールでリモートエージェントを定期実行するビルトインスキル。PCを閉じていてもクラウドで定期的にClaudeが動く。

- **Day 30: /keybindings-help + 総まとめ**
  キーボードショートカットのカスタマイズを学ぶビルトインスキル。そして30日間の振り返り：よく使うコマンドTOP5を決め、自分の作業フローに組み込む。
