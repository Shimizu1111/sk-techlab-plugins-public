# /threads-draft - Threads投稿下書き生成

投稿ネタから投稿ルールに沿った下書きを複数パターン生成し、ストック or 投稿済み管理まで行う。X版の同時生成にも対応。


**投稿ルールの詳細は `references/posting-rules.md` を参照。下書き生成前に必ず読み込むこと。**

## 引数

- `$ARGUMENTS` にテーマやファイルパスを渡せる（省略可）

## 実行手順

### Step 0: アカウント選択

`SNS/` 配下のフォルダ一覧を取得し、ユーザーに選んでもらう。

`$ARGUMENTS` にアカウント名やファイルパスが含まれていれば自動で特定してスキップ。

### Step 1: 投稿ルール・アカウント設計の読み込み

選択されたアカウントの以下のファイルを読み込む:

- `アカウント設計.md`（ターゲット・ポジショニング・トーン）
- `投稿案_*.md`（既存の投稿案があればトーンの参考として）

**読み込んだルールは以降のすべての下書き生成で厳密に適用する。**

### Step 2: 投稿ネタ選択

`$ARGUMENTS` でファイルパスが指定されていればそのファイルを使う。

指定がなければ:
1. `SNS/{アカウント名}/投稿ネタ/` 配下のファイルを一覧表示する
2. 各ファイルのフロントマター（status, priority）を読み取り、一覧に表示する
3. `status: idea` のものを優先的に上に表示する
4. ユーザーに番号で選んでもらう（複数選択も可）

### Step 2.5: 自分の経験ヒアリング（海外事例・外部ソースの場合）

ネタファイルの `source` が `interview` 以外（海外事例・ニュース・Tips等）の場合、下書き生成前にユーザーに聞く:

**「このネタに関連して、自分やクライアントで似た経験ある？ あれば投稿に織り込みたい。」**

- 回答があれば → その経験を投稿の軸にし、海外事例は補強材料として使う
- 「特にない」→ 「自分もこれから試す」「うちでも検討中」等、発信者の立場を最低1文入れる形で生成する
- いずれにせよ、**「誰が書いても同じ」投稿にならないようにする**

### Step 3: 下書き生成（複数パターン）

選択されたネタファイルを読み込み、**2〜3パターンの下書き**を生成する。
各パターンは異なるフックの型を使い、切り口を変える。

#### ソース参照・検証（自動・必須）

ネタファイルにソースURL（`source_urls` フロントマター、`## ソース` セクション、または本文中のURL）がある場合:

**Step A: ソースの一次/二次判定**
1. WebFetch で各URLの内容を取得する（アクセスできない場合はスキップ）
2. そのURLが**一次ソース（オリジナル）か二次ソース（引用・まとめ）かを判定する**
3. 二次ソースの場合、記事内の引用元を辿って一次ソースを特定する
4. 一次ソースが見つかった場合:
   - ネタファイルの `source_urls` を一次ソースに差し替える
   - 投稿内のURLも一次ソースに差し替える
   - ストック（後で語る一覧.md）のソース記載も差し替える

**Step B: 一次ソースの内容取得**
1. 一次ソースをWebFetchで取得し、具体的なデータ・エピソード・数字・文脈を把握する
2. ネタファイルの要約だけでなく、元ソースの生の情報を活かしてリアルな投稿にする

#### 参考URLの投稿内掲載（Step 3の必須判断）

下書き生成時に、以下の判断を**必ず行う**（スキップ不可）:

**判断: このネタにソースURLがある場合、投稿に含めるべきか？**

含める:
- 事故・事件・実話系 → 読者が「嘘でしょ？」と疑う。URLが証拠になる
- 研究データ・公式発表 → 出典があることで説得力が大きく変わる
- 元記事自体が面白い → 読者にも読んでほしい

含めない:
- 自分の実感・実践知として語る投稿（引用戦略B）
- URLを入れると「まとめ記事」感が出る場合
- 複数ソースの横断知見で、特定の1つを出す意味が薄い場合

**含める場合の掲載位置:**
- **1投稿目には絶対に入れない。** 1投稿目はフックと「続きを読ませる」ことだけに集中する。URLがあると流し読みされて離脱する
- **2投稿目以降に入れる。** 本文の流れの中で自然な位置に置く

**上記「投稿ルール」セクションの全ルールに従って下書きを生成する。** 特に以下を意識:

- すべり台原則: 各文が次の文を読ませているか
- 構成フレームワーク: PAS / BAB / ストーリー型のいずれかを使う
- 改行: 意味ブロック方式（同じ文脈は繋げ、転換で空行）
- フック: パターンごとに異なるフックの型を使う
- 着地点: アカウント設計のターゲットの関心事に着地させる

### Step 4: パターン提示

下書きを以下のフォーマットで提示する:

```
## パターンA: {フックの型}

**1投稿目（{文字数}文字）:**

{本文}

**2投稿目（{文字数}文字）:**

{本文}

---

## パターンB: {フックの型}

**1投稿目（{文字数}文字）:**

{本文}

**2投稿目（{文字数}文字）:**

{本文}

---

## パターンC: {フックの型}（あれば）

...
```

提示後、ユーザーに聞く:
**「どのパターンが良い？そのまま使う or 修正したい部分ある？」**

### Step 5: フィードバック対応

ユーザーからのフィードバックに応じて修正する。

- 「もっとカジュアルに」「もっと攻めて」等のトーン調整
- フックの差し替え
- 文字数の調整
- 内容の追加・削除
- パターンの組み合わせ（「Aのフック + Bの本文」等）

納得いくまで何度でも修正する。

### Step 6: ストック or 投稿確定

ユーザーが「OK」「これでいこう」等と言ったら、次を聞く:

**「ストックしておく？それとも投稿済みにする？」**

#### 6a: ストック（後で投稿する場合）

1. **投稿案ファイルに保存**
   - 保存先: `SNS/{アカウント名}/投稿案_{YYYYMMDD}.md`
   - 同じ日付のファイルが既にある場合は末尾に追記する
   - 各投稿に `**status: stock**` を付与する

2. **元の投稿ネタファイルのステータスを更新**
   - `status: idea` → `status: drafted`
   - `posted_date:` はそのまま（空欄のまま）

3. **投稿済みフォルダには移動しない**（ネタフォルダに残す）

4. **Step 7（X版生成）に進む**

#### 6b: 投稿済みにする場合

1. **投稿案ファイルに保存**
   - 保存先: `SNS/{アカウント名}/投稿案_{YYYYMMDD}.md`
   - 同じ日付のファイルが既にある場合は末尾に追記する

2. **元の投稿ネタファイルのステータスを更新**
   - `status: idea` or `status: drafted` → `status: posted`
   - `posted_date:` → `posted_date: {YYYY-MM-DD}`

3. **投稿済みフォルダに移動**
   - 移動先: `SNS/{アカウント名}/投稿済み/`（フォルダがなければ作成）
   - 元の `投稿ネタ/` から `投稿済み/` にファイルを移動する

4. **Step 7（X版生成）に進む**

#### 完了メッセージ（6a・6b共通）
- 保存先のファイルパスを表示
- 残りの未投稿ネタ（`status: idea`）の件数をリマインド

### Step 7: X版の同時生成（オプション）

Threads下書きの保存後、ユーザーに聞く:

**「X版も一緒に作る？」**

ユーザーが希望した場合は、`/x-draft` スキルを呼び出す。

- 同じネタファイルとアカウントを引き継ぐ
- `/x-draft` が X投稿ルールに基づいて独自に下書きを生成する（Threadsの単純変換ではない）
- ストック/投稿済みの管理も `/x-draft` 側で行う

### 投稿案ファイルのフォーマット

```markdown
# Threads投稿案（{YYYY/MM/DD}作成）

アカウント設計: [[アカウント設計]]

---

## 投稿{N}【Threads】: {型}型（元ネタ: {ネタファイル名}）

**status: stock** ← or 省略（投稿済みの場合）

**1投稿目:**
{本文}

**2投稿目:**
{本文}

---

# X投稿案（{YYYY/MM/DD}作成）

ルール: [[X投稿ルール]]
アカウント設計: [[アカウント設計]]

---

## 投稿{N}【X】: {型}型・スレッド（元ネタ: {ネタファイル名}）

**status: stock**

**1ツイート目:**
{本文}

**2ツイート目:**
{本文}
```

### ネタファイルのステータス遷移

```
idea → drafted（下書き完成・ストック中）→ posted（投稿済み）
```

- `drafted`: 下書きが投稿案ファイルにストックされている状態。ネタフォルダに残る
- `posted`: 実際に投稿済み。投稿済みフォルダに移動される

### 投稿前チェック（下書き生成時に自動で確認）

- [ ] 1投稿目は「続きを読ませる」ことに特化しているか
- [ ] 2投稿目以降は200〜450文字か
- [ ] スレッドは2〜3投稿か
- [ ] 冒頭でスクロールが止まるか
- [ ] 改行スタイルは意味ブロックになっているか（同じ文脈は繋がっているか、転換で空行を入れているか）
- [ ] 最終投稿に問いかけ or CTA があるか
- [ ] 全部語りきらず余白があるか
- [ ] ターゲットの関心事に着地しているか
- [ ] 専門用語を使っていないか
- [ ] タグはつけていないか
- [ ] 「○○向け」と直接ターゲットを呼んでいないか
- [ ] ソースURLの掲載判断をしたか（事故・研究等の「嘘でしょ？」系ネタにURLが入っているか）
- [ ] **「自分/うちでは」の一人称要素が入っているか？**（海外事例の翻訳だけで終わっていないか。発信者の存在感がある投稿になっているか）

## 使用例

```
/threads-draft
→ アカウント選択 → ネタ一覧 → 選択 → 複数パターン生成 → 選択 → ストック or 投稿済み → X版も作る？

/threads-draft SNS/AI将軍/投稿ネタ/260525-ChatGPTとClaudeCodeの違い.md
→ 指定ファイルから直接下書き生成

/threads-draft AI将軍
→ アカウント指定 → ネタ一覧 → 選択 → 下書き生成
```

### 典型的なフロー

```
ネタ選択 → Threads下書き生成 → パターン選択 → ストック保存
  → X版も作る？ → X版生成 → ストック保存 → 完了
  → 後日投稿したら /threads-draft で status: posted に変更
```
