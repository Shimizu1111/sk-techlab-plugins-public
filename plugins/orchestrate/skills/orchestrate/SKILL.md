# /orchestrate - マルチエージェントオーケストレーター

ユーザーの要求を複数タスクに分解し、最適なAI CLIエージェント（Claude Code / Codex / Gemini CLI）に振り分けて並行実行するスキル。
**コンテキスト管理を最重視し、長期タスクでも安定して動作する。**

## 起動条件
ユーザーが `/orchestrate` と入力したとき、または「タスクを振り分けて」「並行で進めて」等のオーケストレーション指示があったとき。

## コンテキスト管理戦略（最重要）

このスキルの核心はコンテキスト管理。長期タスクを安定して完遂するために以下を徹底する。

### 進捗ファイルによる外部記憶
プランファイルと同じディレクトリに進捗ファイルを作成・更新し、コンテキストが圧縮されても状態を失わない:

ファイル: `.claude/plans/{plan_name}_progress.md`

```markdown
# 進捗: {ゴール}
最終更新: {timestamp}

## 完了タスク
- [x] #1 JWT調査 (gemini) → 結果: RS256推奨、有効期限15分+リフレッシュトークン
- [x] #2 ログインAPI (codex) → 完了、src/api/auth.ts に実装

## 進行中
- [ ] #3 フロントエンド (codex) → 実行中

## 未着手
- [ ] #4 環境変数更新 (claude)

## 重要な判断・発見
- RS256を採用（Gemini調査結果に基づく）
- DBスキーマにrefresh_tokensテーブル追加済み

## 次にやること
タスク#3の完了を待ち、#4を実行
```

### /compact タイミング
以下のタイミングで `/compact` を実行してコンテキストを圧縮する:

1. **タスク完了ごと**: 各タスクが完了したら、結果を進捗ファイルに書き込み、`/compact` を実行
2. **サブエージェント出力受け取り後**: 長い出力を要約して進捗ファイルに記録し、出力自体はコンテキストから捨てる
3. **3タスク消化ごと**: タスク数が多い場合、3タスクごとに強制 `/compact`
4. **コンテキストが重いと感じたら**: 判断に迷ったら `/compact` する（やりすぎてもコストは低い）

### `/compact` 前に必ずやること
`/compact` する前に以下を進捗ファイルに書き出す（コンテキスト外部化）:
- 完了タスクの結果サマリ
- 進行中タスクの状態
- 重要な判断・発見事項
- 次にやるべきこと

### `/compact` 後に必ずやること
`/compact` 後は進捗ファイルを読み直して状態を復元する:
```
Read .claude/plans/{plan_name}_progress.md
→ 現在の状態を把握
→ 次のタスクを実行
```

### サブエージェントへのコンテキスト渡しルール
- プロジェクト全体のコンテキストを渡さない
- タスクの `context_summary` + 関連ファイルパスのみ渡す
- 「必要なファイルは自分で読んで確認して」と指示する
- サブエージェントの出力は全文保持せず、要点だけ進捗ファイルに記録

## 実行フロー

### Phase 0: プランファイル読み込み（必須）

**プランファイルがなければ実行しない。`/plan` で先に作成するよう案内する。**

1. `.claude/plans/` 内の `*.json` を一覧表示
2. 複数あればユーザーに選択させる、1つなら確認して使用
3. **ファイルがない場合:**
   ```
   プランファイルがありません。
   先に `/plan` でタスクリストを作成してください。
   ```
   → ここで終了
4. `_progress.md` ファイルがあれば途中再開（完了済みタスクはスキップ）

### Phase 2: エージェント振り分けルール

| タスクタイプ | 優先エージェント | フォールバック |
|---|---|---|
| `implementation`（複雑な実装、新機能、大規模リファクタ） | Codex | Claude Code |
| `research`（調査、比較、ドキュメント読解） | Gemini CLI | Claude Code |
| `general`（簡単な修正、設定変更、その他） | Claude Code | - |

**フォールバック条件:**
- Codex/Gemini がエラーを返した場合（レートリミット含む）
- CLI が利用不可な場合
- タスクが失敗して再試行が必要な場合

### Phase 3: 実行ループ

```
for each task (依存関係順):
  1. 進捗ファイルで状態確認（完了済みならスキップ）
  2. タスクのプロンプトを組み立て（context_summary + 指示）
  3. サブエージェント実行（Bash でバックグラウンド or フォアグラウンド）
  4. 出力を受け取り、要点を進捗ファイルに記録
  5. タスクステータスを更新（done / failed）— 進捗ファイルとJSONファイル両方を更新
  6. 失敗なら → フォールバックエージェントで再実行
  7. git diff で変更を確認、必要ならコミット
  8. /compact 判定（上記ルール参照）→ 実行
  9. 次のタスクへ
```

各エージェントへの指示フォーマット:

```
## タスク
{タスクのタイトルと詳細}

## コンテキスト
{プロジェクト構成の要約（1-3行）}
{関連ファイルのパスと要点（全文ではなく要約）}
{前のタスクの結果で必要な情報}

## 制約
- 作業ディレクトリ: {cwd}
- 変更対象ファイル: {対象ファイルリスト}
- テスト: {必要ならテストコマンド}
- 必要なファイルは自分で読んで内容を確認すること
```

**CLI実行コマンド:**

```bash
# Claude Code（ヘッドレス）
claude --print --dangerously-skip-permissions -p "PROMPT" --allowedTools "Bash Edit Write Read Glob Grep"

# Codex（ヘッドレス） ※ stdinを閉じて実行
echo "" | codex exec "PROMPT"

# Gemini CLI（ヘッドレス）
gemini -p "PROMPT"
```

### Phase 4: 監視と統合
1. 各サブエージェントの出力を確認（長い出力は要約のみ保持）
2. 失敗したタスクはフォールバックエージェントで再実行
3. 全タスク完了後、`git diff` で全変更をレビュー
4. コンフリクトがあれば解消
5. 進捗ファイルに最終レポートを書き、ユーザーに報告

## JSONファイルのステータス同期（必須）

タスクの状態変更時は **必ずJSONファイルも更新する**:
- タスク開始時: `"status": "in_progress"`
- タスク完了時: `"status": "done"`
- タスク失敗時: `"status": "failed"`
- 全タスク完了時: JSONのトップレベルに `"completed": "{ISO timestamp}"` を追加

これにより `/plan` で過去のプランの実行状態が確認できる。

## 注意事項
- `--dangerously-skip-permissions` は Claude Code サブエージェントにのみ使用（メインのClaude Codeが監督しているため）
- Codex/Gemini がレートリミットに達したら、残タスクを全て Claude Code にフォールバック
- 依存関係のあるタスクは順序を守る
- 各サブエージェントの出力が長い場合は要約してユーザーに報告
- **コンテキストが不安なら迷わず `/compact`。進捗ファイルがあるので情報は失われない**

## 実行例

ユーザー: 「認証機能を追加して。JWTトークン、ログインAPI、フロントのログインページが必要」

→ タスク分解:
1. [research/gemini] JWT認証のベストプラクティスを調査
2. [implementation/codex] ログインAPIエンドポイントの実装（1に依存）
3. [implementation/codex] JWTトークン生成・検証ミドルウェアの実装（1に依存）
4. [implementation/codex] フロントエンドログインページの実装（2,3に依存）
5. [general/claude] 環境変数とREADMEの更新（4に依存）

→ 実行中のコンテキスト管理:
- タスク1完了 → 結果を進捗ファイルに記録 → /compact
- タスク2,3を並行実行 → 結果記録 → /compact
- タスク4実行 → 結果記録 → /compact
- タスク5実行 → 最終レポート
