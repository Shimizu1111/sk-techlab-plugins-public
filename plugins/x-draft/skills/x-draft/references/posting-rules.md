## 投稿ルール

### 最上位原則: 密度弾（Density Bullet）

**140文字に情報と感情を圧縮し、一撃でタイムラインを貫通させる。**

140文字の制約の中に**フック・価値・余韻**を凝縮する技術。

#### 密度弾を作る4つの武器

**1. 不完全フック**
結論を言わない。答えの手前で止める。

```
x Claude Codeで月次レポートが30分で終わるようになりました。便利です。
  → 完結している。リプもRTもする理由がない

o 毎月丸一日かけてた作業が、ある日1つ変えただけで30分になった。
  → 「何を変えたの？」が開く
```

**2. 断言カット**
短い文で断言する。修飾語を削ぎ落とす。

```
x 個人的にはAI導入がおすすめかなと思っています
  → 弱い。流れる

o AI導入、やるなら今です。
  半年後には当たり前になる。
  → 断言。反論も共感も生まれる
```

ポイント:
- 「~と思います」「~かもしれません」を排除
- 体言止めを多用
- 1文は20文字前後（スマホの1行に収まる）

**3. 面積支配**
改行・空行で物理的にタイムラインの面積を支配する。

```
x AI導入したら月次レポートが30分で終わった。使ったのはClaude Codeだけ。社員増やしてない。
  → 1行に詰め込み。面積が小さく流される

o 月次レポート、30分で終わった。

  使ったのはClaude Codeだけ。

  社員は増やしてない。
  → 空行で面積を支配。タイムラインで目立つ
```

**4. 余白の引力**
全部語りきらない。リプライを誘発する余白を残す。

```
x 毎月3時間やってた作業。Claude Codeに投げたら30分で終わった。CSVを渡してコマンドを実行するだけ。月3時間の削減。
  → 全部言い切った。リプする余地がない

o 毎月3時間やってた作業。

  Claude Codeに投げたら30分で終わった。

  やり方はシンプル。でもほとんどの人がやってない。
  → 「やり方教えて」のリプが来る
```

### 0.3秒テスト

**タイムラインで見た0.3秒で指が止まるか？ 止まらなければその投稿は存在しないのと同じ。**

Xでは冒頭2~3行しかタイムラインに表示されない。
**「0.3秒で指が止まるか？」** を投稿前に必ずチェックする。

### 改行スタイル: 意味ブロック + 面積支配

**同じ文脈の文は繋げ、文脈が変わるところで空行を入れる。面積支配と共存させる。**

- **同じ文脈** → 改行のみ（空行なし）で繋げる
- **文脈の転換** → 空行（1行空け）で区切る
- 「。」のたびに必ず空行を入れる必要はない。文脈が同じなら繋げた方が読みやすい
- **1行は20文字前後**（スマホの1行に収まる）
- 体言止めを積極的に使う
- **面積支配の目的**: タイムラインで物理的に面積を取って目立たせること。ただし「全文空行」ではなく「適切な空行の配置」で実現する

例:
```
でも正直に言うと              <- 同じ文脈だから改行のみ
原因は集客じゃないです。

売れない理由は                <- 文脈が変わったから空行
ほぼ100% 導線です。
```

### 投稿形式

#### 単発投稿（メイン）
- **100~140文字。ギリギリまで使う**
- フック → 価値 → 余白 の3段構成

#### スレッド（週1~2回）
- ステップ解説、Before/After、比較に使う
- 1ツイート目がすべて。ここで止まらなければ全滅
- 各ツイートが独立して「なるほど」と思える密度を持たせる

**スレッドの1ツイート目（親投稿）:**
- 続きのスレッドを読ませることに特化する。内容を詰め込まず、フックと「続きが気になる」導線に集中
- 文字数は少なくてOK（短くても惹きつければ十分）

**2ツイート目以降の密度ルール（最重要）:**
- **各ツイートは140文字ギリギリまで詰め込む。** 薄いツイートを量産するより、密度の高いツイートを少数にする
- 2ツイートで収まるなら2ツイートで終わらせる。無理に引き伸ばさない
- 3ツイート目は必要な場合のみ追加する（内容が溢れる場合）
- **「このツイート単体で読んでも価値があるか？」を各ツイートで確認する。** 価値がないなら前のツイートに統合する

#### 「さらに表示」の折り返し設計

無料プラン（140文字）でも、改行が多いと途中で「さらに表示」が出る。
**この折り返しポイントの直前に仕掛けを置く。**

タイムラインに表示されるのは冒頭2~3行（約40~60文字）。
残りは「さらに表示」の向こう側。

**テクニック1: 結論を匂わせて隠す**
```
Claude Codeで一番やっちゃいけない使い方、知ってますか。

答えは意外すぎて--
                              <- ここで折り返し
（「さらに表示」の向こうに答え）
```

**テクニック2: ストーリーのピーク手前で切る**
```
社員5人の会社の社長に見せた時の話。

「パソコン苦手です」って言われた。

目の前でやってみたら--
                              <- ここで折り返し
（結果は「さらに表示」の向こう）
```

フックを冒頭40文字に収めて、続きが気になる状態で折り返す。

### 投稿の構成パターン

140文字に圧縮する3フレームワーク。

#### PAS（Problem → Agitation → Solution）

```
毎月15日になると胃が痛い経営者、多いと思います。

月次レポート、丸一日かかってませんか。

Claude Codeに投げたら30分で終わりました。

やり方はシンプルです。
```

#### BAB（Before → After → Bridge）

```
月次レポート、去年まで丸2日。

今は30分。しかも人間より正確。

やったことは1つだけ。

Claude Codeにデータを渡しただけです。
```

#### ストーリー型（スレッド1ツイート目）

```
社員5人の会社の社長にClaude Codeを見せた時の話。

「パソコン苦手です」って最初に言われた。

大丈夫ですよ、と言って目の前でやってみた。

結果--
```
→ 2ツイート目で結果を語る

### フック（1ツイート目の冒頭）のルール

**冒頭10~20文字が勝負。**

#### フックの6つの型

| 型 | X用の実例 |
|---|---|
| 恐怖+好奇心 | 「取引先がAIで見積書5分で出してきた時、あなたの会社はどうしますか」 |
| 自己否定 | 「エンジニアやってます。でもエンジニアの価値は来年半分になると思ってます」 |
| 常識の否定 | 「『AIわからない』って言う人、本当の問題はそこじゃないです」 |
| 数字+リスト予告 | 「Claude Codeで実際に消えた業務を全部晒します」 |
| 体験談導入 | 「社員5人の会社の社長にClaude Codeを見せた時の話」 |
| 問いかけ（共感型） | 「毎月の締め日、経理に丸一日取られてませんか」 |

#### X固有の注意
- **冒頭2~3行しか表示されない。** フックはさらに短くする
- **1行目で結論を全部言い切らない**（最重要ルール）
- 140文字では橋渡しフレーズ（バケツリレー）は省略。直接つなぐ

#### 絶対やらない冒頭
- 「おはようございます」「こんにちは」
- 「今日の投稿は~」「報告です」
- 前置きや説明から入る
- 1行目で結論を全部言い切る
- 「~だと思います」「~かもしれません」

### CTA（行動喚起）のルール

#### Xのアルゴリズム上の重み

| アクション | 重み | 優先度 |
|---|---|---|
| リプライ連鎖（会話成立） | x150 | **最優先** |
| 投稿者のリプ返し | +75 | 投稿後15~30分以内に必ず |
| RT | x20 | |
| 引用RT | x15 | |
| リプライ | x13.5 | |
| ブックマーク | x10 | |
| いいね | x0.5~1 | 最も軽い。CTA対象にしない |

#### CTAの3パターン

1. **問いかけ（リプ誘導 = 最重要）**: 「あなたの会社だと何が自動化できそうですか？」
2. **フォローCTA**: 「こういうAI活用の実例を出していくのでフォローどうぞ」
3. **リンク誘導**: 「詳しくはプロフのリンクから」（**本文にリンクは絶対入れない**）

#### リプライ誘導パターン（X固有で強化）
- **二択質問**: 「AとB、どっちが先にやるべきだと思いますか？」
- **体験共有**: 「同じ経験ある人いますか？」
- **不完全情報**: 「やり方はシンプル。わかる人いますか？」

### ナビゲーション要素の使い方

**指差し・矢印はフックの代替ではなく、導線のナビゲーション。**

根拠: 「続きはリプ欄へ」形式で49.5万impを獲得した実例あり。大滝昇平の74万imp投稿もナビゲーションで誘導。

#### 使う場面

| 場面 | 例 | 理由 |
|---|---|---|
| **リプにリンクがある時** | 「詳しくはこちら」+ナビ | リンクを本文に入れずにリプへ誘導。アルゴリズムペナルティ回避 |
| **リプに補足情報がある時** | 「続きはリプ欄へ」 | リプ欄に誘導 = リプ数が増える = アルゴリズム有利 |
| **スレッドの最終ツイート** | 「まとめはこちら」+ナビ | CTAとしてリプに行動を置く |

#### 使わない場面

| 場面 | 理由 |
|---|---|
| **スレッドの1ツイート目の末尾** | フックが弱い時の補助になってしまう。フックの力だけで2ツイート目を開かせる |
| **毎回の投稿** | スパム感が出る。リプに誘導する具体的な理由がある時だけ使う |

### セルフリプ活用

**リンク・補足・CTAはセルフリプ（自分の投稿への返信）に置く。**

Xでは本文にリンクを入れるとリーチが50~90%減る。
セルフリプに置けばペナルティなし + リプ数が増えてアルゴリズム上も有利。

#### セルフリプに置くもの

1. **外部リンク**: 記事URL、体験会申込リンク、YouTube等
   - 本文: 「詳しくはリプに置きました」
   - セルフリプ: URL

2. **補足情報**: 本文140文字に入りきらなかった具体例やデータ
   - 本文: 「やり方はシンプルです」
   - セルフリプ: 「具体的には~」

3. **CTA**: フォロー誘導、体験会案内
   - 本文は価値提供に全振り
   - セルフリプ: 「こういう発信してるのでフォローどうぞ」「体験会やってます→プロフのリンクから」

#### 運用ルール
- セルフリプは**投稿直後に追加する**（時間が空くと別投稿として扱われる）
- スレッド末尾にリンクを置くと単発リンクより**35%クリック増**

### 引用RTされる設計（70/30ルール）

**引用RT（x15）は「70%共感・30%反論の余地」で設計する。**

完全に正しい主張はRTされるが引用RTはされない。
少し「いや、でも...」と思わせる部分を残すと、自分の意見を添えて引用RTしたくなる。

#### 引用RTを生む3パターン（炎上しない形）

**1. 自分の経験から断言する**
```
自分はClaude Code + Obsidianで壁打ちしてます。

月50万のコンサルと同じ質の打ち手が出てくる。

もちろん人によって合う合わないはある。
```
→ 「自分はこう」と引用RTで経験を語りたくなる

**2. 二項対立を提示する**
```
ChatGPTとClaude Code、用途が違う。

相談相手→ChatGPT
実装パートナー→Claude Code

使い分けてる人いますか？
```
→ 「自分はこう使い分けてる」と引用RTで語りたくなる

**3. 問いかけで開く**
```
AI社員、いきなり作るのが一番の失敗パターンだと思ってる。

でも人によって正解は違うかもしれない。

うまくいった人、どうやった？
```
→ 「自分の場合は~」と引用RTで反論・共感が集まる

#### 注意
- **他社・他サービスを否定する形の引用RT誘発はやらない**
- 「○○は死んだ」「○○は詐欺」は炎上直行
- 比較はOK、否定はNG（炎上防止3原則と同じ）

### フォーマットルール

| 項目 | ルール |
|---|---|
| タグ | **つけない** |
| 外部リンク | **本文に絶対入れない**（リーチ50~90%減。リプかプロフに置く） |
| 画像 | テキスト+画像が強い。スレッドでは3~4ツイートごとに画像推奨 |
| 絵文字 | 控えめ。ポイント的に |
| トーン | ベースは「です・ます」。要所だけ「だ・である」で断定。ちゃんとした言葉遣いを保つ（セミナー集客導線のため信頼感重視） |

### 固定ツイート戦略

**プロフィールを訪問した人が最初に見るツイート。体験会誘導の最重要導線。**

#### 固定ツイートに置くべき内容

以下のいずれか1つを固定する:

1. **自己紹介+実績+CTA型**（最推奨）
```
エンジニア歴○年。
今はClaude Codeで中小企業の業務を自動化してます。

実績:
・月次レポート 丸1日→30分
・請求書作成 毎月3時間→5分
・議事録整理 毎週2時間→0分

無料体験会やってます。
詳しくはプロフのリンクから。
```

2. **バズった投稿をそのまま固定**
   - 過去に最もエンゲージメントが高かった投稿を固定する
   - 「この人の投稿は面白い」と思わせてフォローに繋げる

3. **体験会告知を直接固定**
   - 開催が近い時期に差し替える

#### 運用ルール
- **月1回は見直す**（古い情報が固定されたままにしない）
- バズった投稿が出たらすぐ差し替えを検討する
- 体験会の日程が変わったら即更新

### 頻度・タイミング

- 1日1~3投稿
- 最低ライン: 週3回
- **最適時間帯: 朝6~8時 / 昼12~13時 / 夜20~22時**（夜がリーチ最大）
- 木曜夜がベスト。火~木が安定
- **投稿後15~30分はXに留まり、来たリプ全てに返信する**
- 連続投稿は最低30分間隔を空ける（スパム判定回避）

#### 投稿後の行動ルーティン

投稿したら終わりではない。**投稿後30分の行動がバズるかどうかを決める。**

**投稿直後~30分:**
1. Xに留まる（アプリを閉じない）
2. 来たリプライ全てに返信する（会話成立 = x150ブースト）
3. 自分の投稿にセルフリプで補足・リンクを追加する

**投稿後30分~1時間:**
4. ターゲット層（経営者・AI活用者）の投稿にリプライしに行く（3~5件）
5. 関連投稿にいいねを押す（1時間に100いいね以下。超えるとスパム判定）

**毎日:**
6. 自分のフォロワーの投稿に1日5件以上リプライする（関係性を維持）
7. 新しいアカウントを1日3~5件フォローする（急激なフォロー増はNG）

### アルゴリズムの核心

- Xは「リプライ連鎖」が最強シグナル（x150）
- **投稿後15~30分の初速が全て**
- Grok AIがコンテンツを意味レベルで評価。フォロワー数よりも内容の質が重要
- 質の高いコンテンツは少フォロワーでも「おすすめ」に載る
- **リプに返信して「会話成立」させることがバズの最大条件**

### 炎上防止ルール（最重要）

**拡散力が強い分、炎上リスクも高い。**
**「強い主張」と「攻撃」は違う。断言はしても、人や他サービスを貶めない。**

#### 3原則

1. **他者・他サービスを否定しない**
   - 「ChatGPTは使えない」→ NG
   - 「ChatGPTも便利。でも蓄積型のAIを知ったら世界変わった」→ OK
   - 比較はOK、否定はNG

2. **読者を見下さない**
   - 「知らない人は損してます」→ ギリOK
   - 「知らない人はヤバい」→ NG
   - 無知を責めず、知る機会を提供する

3. **属性を攻撃しない**
   - 「コンサルは全員ボッタクリ」→ NG
   - 「見極め方を教えます」→ OK
   - 特定の個人・企業・業界を名指しで否定しない

**命令口調の禁止**: 「しろ」「するな」等の命令形は使わない。「してください」「してみてください」は可。断定調（「~です」「~じゃない」）はOKだが、読者への命令は炎上リスクがある。

#### ネガティブシグナルの重さ

| トリガー | ペナルティ |
|---|---|
| ユーザーからの通報 | -369倍 |
| ブロック・ミュート | -74倍 |
| 攻撃的なトーン | Grokが検知し配信80%カット |
| 「興味なし」を押される | そのユーザーへの表示が140日間80%減 |

**1件の通報 = いいね369件分で相殺が必要。** 炎上は絶対に割に合わない。

### やってはいけないこと

共通:
- 「いいねしてね」「フォローしてね」の直接依頼（ペナルティ）
- AI生成文を無加工で大量投稿
- 短時間の大量フォロー・アンフォロー
- 挨拶から始める
- 記事っぽい丁寧な構成にする
- 全部語りきって余白をゼロにする
- 1行目で結論を全部出す

X固有:
- **外部リンクを本文に入れる**（リーチ50~90%減。最大のペナルティ）
- **ネガティブ・攻撃的なトーン**（Grokが検知。配信80%カット）
- **同一内容の繰り返し投稿**（スパム判定）
- **連続投稿**（最低30分は間隔を空ける）
- **リプライに返信しない**（最強シグナル+75の放棄）
- **10ツイート超のスレッド**（長すぎて離脱）

### 経営者に刺さる内容の原則

経営者の関心は **売上・コスト・人材・時間** の4つだけ。

- 「新機能が出た」→「請求書が5分で終わる」
- 「プロンプト」→「部下への指示と同じ」
- 「API連携」→「社員を増やさずに回せる」

#### 反応するもの
- 数字のBefore/After
- 失敗談（成功より信頼される）
- コスト削減の具体額
- 「競合はもう始めてる」（FOMO）
- 「人を増やさずに回す」

#### 反応しないもの
- 技術の仕組み説明
- 統計データだけの投稿
- 「簡単です」の連発
- 抽象的なモチベーション論

### 暗黙的ターゲティング（最重要）

- 「経営者向け」「社長さんへ」とは絶対に書かない
- **経営者しか興味がない話題を自然に扱う**
- 「社員の仕事が3時間短縮」「月次レポートが自動化」「採用コスト削減」→ 経営者が勝手に反応する

### 海外事例・ソース記事の引用戦略

海外事例（`海外事例/` フォルダ）やソース記事を元に投稿する場合、ソースの出し方を**ネタの種類で使い分ける**。

#### A. 権威を借りる（ソース1つだけ明示）

主語が1つで強い時に使う。「開発者本人が〜」「30人の会社が〜」等。

```
○ 「Claude Codeを作った本人が、設定ファイルを公開した。驚いたのは"たった100行"。」
→ 権威ある1ソースを冒頭に。以降の解釈は自分の見解として語る
```

向いているネタ: 開発者インタビュー、特定企業の導入事例

#### B. 自分の実感として語る（ソース出さない）

一般的なコツ・フレームワーク系。毎日使っている実践者として語る。

```
○ 「AIへの指示、"いい感じにやって"だと失敗する。"こうなったら完成"を伝えるだけで全然違う。」
→ 複数ソースの知見を自分のフィルターで消化した"自分の言葉"
```

向いているネタ: ノウハウ・Tips、使い方のコツ、フレームワーク

#### C. 「海外では〜」で匂わせる（ソース曖昧）

具体的な数字・金額がある事例。

```
○ 「アメリカでは外注費6種類を丸ごとAIに置き換えた会社が出てきてる。」
→ URLは出さない。興味を持った人がリプで聞いてくる導線になる
```

向いているネタ: 海外企業の数字付き事例、コスト削減・売上向上の具体例

#### 判断基準

| ネタの特徴 | パターン |
|---|---|
| 特定の人物・企業名が主語 | A（権威を借りる） |
| 自分も日常的にやっていること | B（自分の実感） |
| 具体的な数字・金額がある海外事例 | C（海外匂わせ） |
| 複数ソースの横断知見 | B（自分の実感） |

#### 絶対やらないこと

- URLを本文に貼る（アルゴリズムペナルティ）
- 複数ソースを列挙する（記事調すぎる）
- 翻訳記事のまとめっぽくする（実践者ポジションと矛盾）
