# /obs-audit-context - プロジェクトのコンテキスト管理を整備する

Obsidian のドキュメントフォルダで Claude Code が適切に動作するための仕組み（CLAUDE.md・`_index.md`・MEMORY・rules）を監査し、不足があれば構築・修正する。

## 前提

- Obsidian Vault 内の各フォルダには大量のドキュメントがある
- Claude は全ドキュメントを毎回読むことはできない
- だから **2段階ルーティング** で必要なドキュメントだけ参照させる設計が必須:
  - **第1段階（CLAUDE.md）**: ユーザーの意図 → 該当フォルダを特定
  - **第2段階（`_index.md`）**: フォルダ内で → 具体的にどのファイルを読むか決定
- CLAUDE.md + `_index.md` + MEMORY + rules が噛み合って初めて、Claude は適切な回答を返せる

## 実行手順

### Step 1: 対象プロジェクトの特定
- CWD を確認し、どのプロジェクトフォルダにいるか特定する
- CLAUDE.md を探す（CWD → 親 → Vault ルート）

### Step 2: 現状把握
- プロジェクト内のファイル・フォルダ構成を把握する
- 既存の CLAUDE.md、`_index.md`、rules、MEMORY.md の該当セクションを読む
- Vault CLAUDE.md の該当コンテキスト参照（`@docs/xxx.md` 等）を確認
- どんなドキュメントがあり、どんな会話が想定されるか把握する

### Step 3: チェック＆修正

#### A. CLAUDE.md（最重要）
CLAUDE.md は「Claude がこのフォルダで仕事するための取扱説明書」。以下を満たすこと:

1. **フォルダの目的**: このフォルダは何のためにあるか（1-2行）
2. **Claude の役割**: このフォルダの文脈で Claude は何をする存在か
3. **常に意識すべきこと**: 回答時に必ず踏まえるべき前提・原則・制約
4. **フォルダルーティング（第1段階）**: ユーザーの意図 → 該当フォルダへの振り分け
5. **ファイル構成**: フォルダ内の構造説明（何がどこにあるか）
6. **150行以内**: 超えたら詳細は別ファイルに切り出す

チェック:
- [ ] CLAUDE.md が存在するか → なければ作成を提案
- [ ] 上記1-5が揃っているか → 不足があれば追記を提案
- [ ] ルーティングに「`_index.md` を必ず読む」指示があるか
- [ ] 肥大化していないか（150行超え）

#### B. 2段階ルーティングの設計

##### 第1段階: CLAUDE.md のルーティングテーブル

ユーザーの意図 → フォルダを特定し、**そのフォルダの `_index.md` を必ずReadする**。

```markdown
## フォルダルーティング（2段階遅延ロード）

**ルーティング手順:**
1. ユーザーの意図を判定
2. 該当フォルダを特定
3. **そのフォルダの `_index.md` を Read する**（必須）
4. `_index.md` の指示に従い、必要なファイルを Read する
5. 作業実行

| ユーザーの意図・キーワード | ルーティング先 |
|---|---|
| 「○○」「△△」 | `フォルダ名/` |
| 「□□」 | `フォルダ名/` |
```

**第1段階のポイント**:
- テーブルは「意図 → フォルダ」の2列でシンプルにする（読むべきファイルは `_index.md` が担当）
- トリガーは「ユーザーが実際に使う言葉・キーワード」で書く（技術用語ではなく口語）
- 複数フォルダにまたがる場合は、主テーマのフォルダを優先しサブで他フォルダも読む

##### 第2段階: 各フォルダの `_index.md`

**`_index.md` とは**: フォルダに入った Claude が「次に何を読むべきか」を即座に判断できるガイド。

**`_index.md` のフォーマット**:
```markdown
# [フォルダ名] インデックス

## このフォルダの役割
[1-2行で説明]

## ファイル一覧と読み分けガイド

| ファイル | 内容 | こういう時に読む |
|---|---|---|
| `ファイル1.md` | ○○の定義 | 「○○って何？」「○○を見直したい」 |
| `ファイル2.md` | △△の計画 | 「次何する？」「計画の進捗は？」 |
| `ファイル3.md` | □□のログ | 「前にやったこと」「振り返り」 |

## 読み分けルール
- 計画系の質問 → `ファイル2.md` を最優先で読む
- 振り返り系 → `ファイル3.md` + `ファイル1.md` を合わせて読む
- ファイルが多い場合 → Glob で一覧してファイル名で絞り込む
- 迷ったら → [最も基本的なファイル] を読めば大抵カバーできる
```

**`_index.md` の設計ルール**:
1. **全ファイルを網羅する**: フォルダ内のファイルが漏れなく載っていること
2. **「こういう時に読む」が具体的**: ユーザーの口語ベースで書く
3. **読み分けルールを明記**: 複数ファイルの組み合わせが必要なケースも書く
4. **ファイル数が不定のフォルダ**: 命名規則とGlobパターンを書く（個別ファイルの列挙は不要）
5. **20行以内を目安**: 長くなるならフォルダ構成を見直すべきサイン

**`_index.md` が必要なフォルダの基準**:
- ファイルが3つ以上あるフォルダ → 必須
- ファイルが2つ以下 → 不要（CLAUDE.md のルーティングテーブルで十分）
- サブフォルダがある場合 → 親フォルダの `_index.md` でサブフォルダの役割も説明する

##### チェック項目

CLAUDE.md ルーティング:
- [ ] 全サブフォルダがルーティングでカバーされているか
- [ ] 「`_index.md` を必ず読む」指示がルーティング手順に含まれているか
- [ ] トリガーが具体的か（「計画の見直し」◯ / 「作業するとき」✕）

`_index.md`:
- [ ] ファイル3つ以上のフォルダに `_index.md` が存在するか
- [ ] 全ファイルが網羅されているか
- [ ] 「こういう時に読む」が具体的・口語的か
- [ ] 読み分けルールがあるか
- [ ] 参照先ファイルが全て実在するか
- [ ] 内容が最新か（フォルダ内のファイル増減と同期しているか）

#### C. MEMORY.md の該当セクション
- [ ] プロジェクトのセクションがあるか → なければ追加を提案
- [ ] CLAUDE.md と矛盾していないか
- [ ] 適切な粒度か（CLAUDE.md に詳細がある場合は要約+参照のみ）
- [ ] MEMORY.md 全体が200行以内か

#### D. Vault CLAUDE.md との連携
- [ ] Vault CLAUDE.md のコンテキスト参照に登録されているか（必要な場合）
- [ ] `docs/` 内に関連コンテキストファイルがあるなら、内容が実態と一致しているか

#### E. rules（.claude/rules）
- [ ] プロジェクト固有のルールが必要なら設定されているか
- [ ] 既存ルールが実態と合っているか

### Step 4: レポート出力

```markdown
# コンテキスト管理 監査レポート
対象: [プロジェクト名]（[パス]）
実施日: YYYY-MM-DD

## サマリー
- 合格: X項目
- 警告: X項目
- 要修正: X項目（うち新規作成: X件）

## 要修正（今すぐ直すべき）
1. [問題の説明] → [修正案]

## 警告（近いうちに対応すべき）
1. [問題の説明] → [推奨アクション]

## 合格
- [チェック済み項目のリスト]

## 修正を実行しますか？
```

### Step 5: 承認を得て修正実行
- CLAUDE.md のルーティング修正（2段階化）
- `_index.md` の新規作成・修正
- MEMORY.md の該当セクション更新
- docs/ のコンテキストファイル更新
- 全て修正後、最終確認を出力
