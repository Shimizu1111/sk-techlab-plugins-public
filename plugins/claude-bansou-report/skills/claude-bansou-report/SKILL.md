# セッション受講レポート生成スキル

Nottaの文字起こし（テキスト直貼りまたはURL）から、以下を一気通貫で作成する：

1. 文字起こしの保存
2. 受講内容まとめの作成
3. まとめ画像用プロンプト（3〜6枚）の生成
4. **Codex CLI** で GPT Image 2（ChatGPT Images 2.0）による画像生成
5. 公開用ページへの統合
6. Notionへの同期（セッション記録の作成）

---

## ステップ0: 受講者の特定とフォルダ作成

`$ARGUMENTS` に受講者名やパスが含まれている場合はそれを使う。
なければ `ClaudeCode受講者管理/受講履歴/` 配下のフォルダ一覧を提示して選択させる。

### フォルダ命名規則

受講者フォルダは `{NotionID}_{受講者名}` の形式で管理する。
- 例: `1_荒賀さん`、`2_藤さん`
- NotionのIDは受講者マスタの `auto_increment_id`（自動採番）
- IDがまだ付与されていないフォルダ（例: `横井さん`）は、ステップ6のNotion同期時にリネームする

以降、選択されたパスを `{受講者パス}` と呼ぶ。
例: `ClaudeCode受講者管理/受講履歴/2_藤さん`

受講者名はフォルダ名からIDプレフィックスを除いた部分を使う（`2_藤さん` → `藤さん`）。

### セッションフォルダの作成

セッション日付（YYYYMMDD形式）のフォルダを、テンプレートからコピーして作成する：

```bash
cp -r "ClaudeCode受講者管理/テンプレートフォルダ/" "{受講者パス}/{YYYYMMDD}"
```

テンプレートフォルダには以下のサブフォルダが含まれている：
- `文字起こし/`
- `受講内容まとめ/`
- `受講まとめ画像/`
- `公開用ページ/`

以降、`{受講者パス}/{YYYYMMDD}` を `{セッションパス}` と呼ぶ。
例: `ClaudeCode受講者管理/受講履歴/2_藤さん/20260601`

## ステップ1: 文字起こしの取得

### $ARGUMENTS にNotta URLが含まれている場合
`notta.ai` を含むURLが `$ARGUMENTS` にあれば、入力方法の確認をスキップし、そのまま Playwright で取得に進む。

### それ以外の場合、ユーザーに入力方法を確認：

> 文字起こしをどう取り込みますか？
> 1. **テキスト直貼り** — Nottaからコピーした文字起こしをそのまま貼り付けてください
> 2. **Notta URL** — NottaのURLを貼ってください（Playwrightで取得します）

### テキスト直貼りの場合
そのまま受け取る。

### Notta URLの場合

**重要: NottaはJavaScriptで動的にレンダリングされるため、WebFetchでは取得できない。Playwrightを使う。**

以下のPythonスクリプトをBashで実行する：

```python
import asyncio
import json
from playwright.async_api import async_playwright

async def fetch_notta():
    async with async_playwright() as p:
        browser = await p.chromium.launch(headless=True)
        page = await browser.new_page()

        transcript_data = []

        async def handle_response(response):
            url = response.url
            if 'socket.io' in url and 'record_id' in url:
                try:
                    body = await response.text()
                    if 'init_document' in body and len(body) > 10000:
                        transcript_data.append(body)
                except:
                    pass

        page.on("response", handle_response)

        await page.goto("{NOTTA_URL}", wait_until="domcontentloaded", timeout=60000)
        await page.wait_for_timeout(15000)
        await browser.close()

        if not transcript_data:
            print("ERROR: No transcript data captured")
            return

        raw = transcript_data[0]
        json_start = raw.find('["init_document"')
        if json_start == -1:
            print("ERROR: Could not find init_document")
            return

        data = json.loads(raw[json_start:])
        doc = data[1]["doc"]

        lines = []
        for item in doc["content"]:
            if item.get("type") == "transcribeItem":
                attrs = item.get("attrs", {})
                speaker = attrs.get("speaker", "Unknown")
                empty_attrs = attrs.get("empty_content_attrs", {})
                start_time = empty_attrs.get("start_time", 0)
                minutes = int(start_time // 60)
                seconds = int(start_time % 60)
                timestamp = f"{minutes:02d}:{seconds:02d}"
                text_parts = []
                for content in item.get("content", []):
                    if content.get("type") == "text":
                        text_parts.append(content.get("text", ""))
                full_text = "".join(text_parts)
                if full_text.strip():
                    lines.append(f"{timestamp} {speaker}: {full_text}")

        print("\n\n".join(lines))

asyncio.run(fetch_notta())
```

**仕組み**: NottaはWebSocket経由で文字起こしデータ（JSON）を取得している。Playwrightでページを読み込み、WebSocketレスポンスをインターセプトして `init_document` イベントから全文を抽出する。

Playwrightが使えない場合は、ユーザーにテキスト直貼りを依頼する：
> Playwrightが利用できません。Nottaの画面から文字起こしテキストをコピーして、ここに貼り付けてください。

### セッション日付の確認
文字起こし内容から日付が特定できない場合、ユーザーに確認する：
> このセッションの日付はいつですか？（例: 2026-06-01）

### 保存
`{セッションパス}/文字起こし/{YYYY-MM-DD}.md` に保存する。

```markdown
---
source: {Notta URL or "直接入力"}
date: {YYYY-MM-DD}
type: 受講文字起こし
---

{文字起こし全文}
```

## ステップ2: 受講内容まとめの作成

文字起こし全文を読み、まとめを作成する。

### 全文を読み切る（絶対に省略しない）

文字起こしは最初から最後まで全て読む。2000行を超える場合は分割して読む。

### まとめ作成ルール

#### 1. 具体例・エピソードは個別に全部書き出す
「さまざまな〜」「いくつかの〜」でまとめない。

#### 2. 数字は全て拾う
金額・人数・割合・年数・回数・期間、全て具体的に書く。

#### 3. 印象的な発言はそのまま引用する
`**「〜」**` で話者の口調のまま引用。

#### 4. 主張にはロジック（なぜそう言っているか）もセットで書く
結論だけ書いて「〜と語った」で終わらせない。

#### 5. 手順・フレームワークは具体的な中身まで展開する
「〜が重要」「〜がポイント」で止めない。中身を書く。

#### 6. 実演した部分は詳細に再現する
「実演した」の一言で済ませない。

#### 7. 自分の解釈や意見は加えない

### セクション構成の目安
- 30分〜1時間 → 5〜8セクション
- 1〜2時間 → 8〜12セクション
- 2時間以上 → 12セクション以上

### ボリュームの自己チェック
文字起こし行数に対して、まとめは最低1/3の行数を目指す。

以下に該当したら書き直す：
- 「さまざまな」「いくつかの」「多くの」で具体例をまとめてしまっている
- 数字が出てきたのに形容詞で丸めている
- あるセクションが3行以下で終わっている
- 「〜について語った」「〜と述べた」のメタ記述で中身がない

### 保存
`{セッションパス}/受講内容まとめ/{YYYY-MM-DD}_まとめ.md` に保存する。

**構成ルール: 画像 → 概要 → 詳細の順。画像は全て先頭にまとめて配置する。**
Notta URLがある場合は冒頭に記載する。

```markdown
# {セッションテーマ} まとめ（{YYYY-MM-DD}）

> Notta: {Notta URL}（URLがある場合のみ）

{概要：2-3行}

---

## セッション画像

![[{YYYY-MM-DD}_1.png]]
![[{YYYY-MM-DD}_2.png]]
![[{YYYY-MM-DD}_3.png]]
（生成した全画像を並べる）

---

## 1. {トピック}
...

---

## N. 重要フレーズ集
- **「〜」** — {文脈の補足}
```

**注意**: 画像はステップ4で生成後、このまとめファイルに埋め込みリンクを追記する。

## ステップ3: 画像プロンプトの作成

まとめ内容をもとに、**3〜6枚分の画像プロンプト**を作成する。

### 基本方針

**プロンプトは「何があったか・何を学んだか」の内容まとめを書く。デザイン指示（構図・色・スタイル等）は書かない。**
ビジュアル化は画像生成モデルに完全に任せる。こちらが下手にデザインを指定するより、内容だけ渡した方が良い画像ができる。

### プロンプトの書き方

- **日本語**で書く
- セッションのまとめ内容から、画像にしたいトピックを選ぶ
- そのトピックで**何をやったか、何が学びだったか、どういう結論になったか**を端的にまとめる
- 具体的な発言の引用やキーワードを含めると効果的
- デザイン・レイアウト・色・スタイルの指示は一切書かない

### トピック選定の観点（この中から3〜6個を選ぶ）

1. **セッションの核心メッセージ** — 最も重要な学び・気づき
2. **やったこと** — セッション中に実際に進めた作業や設定
3. **提案されたフロー・仕組み** — 自動化フローや運用設計
4. **業界の議論・洞察** — AI時代の変化など戦略的な話題
5. **次のアクション** — 今後やるべきこと

### プロンプトファイルの保存

`{セッションパス}/受講まとめ画像/{YYYY-MM-DD}_プロンプト.md` に保存する。

```markdown
# 画像生成プロンプト（{YYYY-MM-DD}）

## 画像1: {トピック名}
{そのトピックの内容まとめ。何があったか、何を学んだか。}

---

## 画像2: {トピック名}
{同上}
```

## ステップ4: Codex CLI で GPT Image 2（ChatGPT Images 2.0）による画像生成

**Codex CLI（`codex` コマンド）の組み込み画像生成機能**を使って画像を生成する。
Codex CLI は ChatGPT 経由で **GPT Image 2（ChatGPT Images 2.0）** を利用できる。APIキー不要。
**注意: `gpt-image-1` は旧モデル（API用）であり、GPT Image 2 とは別物。Codex CLI の組み込み機能を使えば自動的に最新の GPT Image 2 が使われる。**

### 起動コマンド

**全画像を必ず1つの `codex exec` セッション内で生成する。**
同一セッション内で生成することで、画像のスタイル・配色・レイアウトが統一される。
Bashツールで以下のように呼び出す（`run_in_background: true` 推奨）：

```bash
codex exec -s danger-full-access "
以下の画像を順番に生成してください。

【重要】全画像を同一のデザインスタイル・カラーパレット・レイアウト体裁で統一してください。
1枚目を生成したら、その雰囲気・配色・フォント・レイアウトを2枚目以降でも必ず踏襲してください。
シリーズものとして一目で同じセッションの画像だとわかるようにしてください。

画像1:
{内容まとめテキスト}

画像2:
{内容まとめテキスト}

...（3〜6枚分）

各画像を生成したら ~/.codex/generated_images/ から以下のパスにコピーしてください：
画像1: {セッションパス}/受講まとめ画像/{YYYY-MM-DD}_1.png
画像2: {セッションパス}/受講まとめ画像/{YYYY-MM-DD}_2.png
...
全完了後に各ファイルのパスとサイズを報告してください。
"
```

### 重要な注意事項

- **`codex exec`** を使う（`codex` 単体はインタラクティブモード）
- サンドボックスは **`-s danger-full-access`** を指定（ファイル書き込みに必要）
- タイムアウトは **600000ms（10分）** に設定する
- Codex は画像を `~/.codex/generated_images/{session_id}/` に生成し、自分で指定パスにコピーする
- **APIキーは不要**（Codex CLI 自身の認証で動作する）
- **Node.jsスクリプトを書く必要はない**（Codex の組み込み機能を使う）

### Codex が使えない場合のフォールバック

`codex` コマンドが見つからない場合：

> Codex CLIがインストールされていません。以下のいずれかで対応してください：
> 1. `npm install -g @openai/codex` でインストールしてから再実行
> 2. プロンプトファイル（`{プロンプトファイルパス}`）を使って、ChatGPT等で手動生成
>
> 画像が準備できたら `{セッションパス}/受講まとめ画像/` に配置して、「画像を配置した」と教えてください。

### 完了確認と画像の一意性チェック

**重要: Codexはコピー時に同一ファイルを全スロットに配置するバグがある。必ず以下の検証を行う。**

Codex実行後、保存先フォルダに画像ファイルが存在し、全て異なるファイルであることを確認する：

```bash
# ファイル存在確認
ls -la "{セッションパス}/受講まとめ画像/{YYYY-MM-DD}_"*.png

# 一意性チェック（md5が全て異なること）
for f in "{セッションパス}/受講まとめ画像/{YYYY-MM-DD}_"*.png; do echo "$(basename "$f"): $(md5 -q "$f")"; done
```

**全ファイルのmd5が同一の場合（= 同じ画像がコピーされている）:**

Codexのセッションディレクトリから個別画像を手動で再配置する：

```bash
# セッションIDを特定
ls -td ~/.codex/generated_images/* | head -n 1

# 生成時刻順にソートして1〜N番にマッピング
SRC=~/.codex/generated_images/{session_id}
DEST="{セッションパス}/受講まとめ画像"
files=($(ls -tr "$SRC"/*.png))
for i in {1..N}; do
  cp "${files[$i]}" "$DEST/{YYYY-MM-DD}_${i}.png"
done
```

- 成功した画像: ファイル名と保存先を記録
- 失敗した画像: エラー内容を報告し、手動生成を案内

## ステップ5: 公開用ページの作成

全てを統合した公開用ページを作成する。

### 保存先
`{セッションパス}/公開用ページ/{YYYY-MM-DD}_レポート.md`

### ページ構成

```markdown
# {セッションテーマ} 受講レポート

> {セッションの概要 1-2行。受講者にとっての価値が伝わるように}

---

## セッションハイライト

### {トピック1のタイトル}

{まとめから要点を抽出・再構成。公開に適した分量に調整する}

![[{YYYY-MM-DD}_1.png]]

---

### {トピック2のタイトル}

{同上}

![[{YYYY-MM-DD}_2.png]]

---

（画像を適切な位置に配置しながら、全トピックをカバーする）

---

## 学びのポイント

- {箇条書きで主要な学びを3〜5個}

## 次のアクション

- [ ] {受講者が次にやるべきこと}
- [ ] ...

---

*このレポートは {YYYY-MM-DD} のセッション内容をもとに作成されました。*
```

### ページ作成のポイント

- **まとめの全文コピーではない**。公開に適した形に再構成する
- 画像は内容の流れに自然に配置する（Obsidian記法 `![[ファイル名]]`）
- 受講者が振り返りに使えるよう、ポイントを明確にする
- 長すぎない（まとめの1/2〜2/3程度の分量が目安）

## ステップ6: Notion同期

セッションレポートの内容をNotionの「セッション記録」データベースに同期する。

### Notion データベース情報

| DB名 | データソースID |
|---|---|
| 受講者マスタ | `collection://353ba38d-b047-8112-93d8-000bb46f3da9` |
| セッション記録 | `collection://353ba38d-b047-8180-a1a5-000b6b8a210e` |

- 受講者マスタの `ID` は `auto_increment_id`（Notionが自動採番）
- セッション記録の `受講者` プロパティは受講者マスタへのリレーション

### 6-1. 受講者の特定（Notion側）

受講者名（フォルダ名からIDプレフィックスを除いた部分）で `notion-search` を実行し、受講者マスタのページURLを取得する。

**受講者が見つかった場合:**
- そのページURLを `{受講者ページURL}` として記録する
- フォルダにIDプレフィックスがなければ、Notionページを `notion-fetch` して `userDefined:ID` を取得し、ローカルフォルダをリネームする

**受講者が見つからない場合:**
受講者マスタに新規作成する。

```
notion-create-pages:
  parent: {"data_source_id": "353ba38d-b047-8112-93d8-000bb46f3da9"}
  properties:
    受講者名: {受講者名}
    ステータス: "受講中"
    date:開始日:start: {セッション日付}
    date:開始日:is_datetime: 0
```

作成後、そのページを `notion-fetch` して自動採番された `userDefined:ID` を取得する。

**ローカルフォルダのリネーム:**
IDプレフィックスがないフォルダ（例: `横井さん`）を `{ID}_{受講者名}`（例: `4_横井さん`）にリネームする。

```bash
mv "ClaudeCode受講者管理/受講履歴/{旧フォルダ名}" "ClaudeCode受講者管理/受講履歴/{ID}_{受講者名}"
```

リネーム後、以降の `{受講者パス}` と `{セッションパス}` も新しいパスに更新する。

### 6-2. セッション情報の確認

以下の情報をユーザーに確認する（まとめ内容から推定できるものは提案する）：

> Notionに同期するセッション情報を確認します：
> - **タイトル**: {推定: 例「第3〜4回 - Git連携」}
> - **回数**: {推定: 例 4}
> - **日付**: {セッション日付}
> - **内容**: {まとめの概要を1行で}
> - **次回の予定**: {まとめから抽出 or 空欄}
> - **理解度**: ◎ / ○ / △（どれにしますか？）
>
> このまま同期してよいですか？修正があれば教えてください。

### 6-3. セッション記録の作成

セッション記録データベースに新規ページを作成する。

```
notion-create-pages:
  parent: {"data_source_id": "353ba38d-b047-8180-a1a5-000b6b8a210e"}
  properties:
    タイトル: {セッションタイトル}
    ステータス: "下書き"
    内容: {セッション内容の要約（1行）}
    受講者: "[\"https://app.notion.com/p/{受講者ページID}\"]"
    回数: {セッション回数（数値）}
    date:日付:start: {YYYY-MM-DD}
    date:日付:is_datetime: 0
    次回の予定: {次回予定テキスト}
    理解度: {◎ / ○ / △}
  content: {受講内容まとめの内容（下記の変換ルール適用後）}
```

**注意**:
- `受講者` プロパティはJSON配列の文字列で渡す。リレーションが正しく設定されると、受講者マスタ側の「セッション記録」にも自動的に表示される
- contentに設定するのは `受講内容まとめ` の内容（公開用ページではない）

### Obsidian → Notion コンテンツ変換ルール

受講内容まとめをNotionに同期する際、以下の変換を行う：

1. **画像埋め込み行の除外**: `![[ファイル名.png]]` 形式の行は除外する（画像はステップ6-4で別途アップロードする）
2. **「セッション画像」の見出しは残す**: `## セッション画像` の見出し自体は残す（画像の挿入位置マーカーとして使用）
3. **それ以外のMarkdown記法はそのまま**: 見出し・太字・引用・リスト等はNotionでもそのまま使える

### 6-4. 画像のアップロード

Notion MCPには画像アップロード機能がないため、Notion APIを直接叩くスクリプト（`.scripts/notion-upload-images.py`）を使用する。

**前提**: `.env` に `NOTION_API_KEY=ntn_xxxxx` が設定されていること。
**前提**: Notionワークスペースにて対象ページ（またはその親ページ）にIntegrationが接続されていること。

`--after-heading "セッション画像"` を指定して、「セッション画像」見出しの直後に画像を挿入する：

```bash
python3 .scripts/notion-upload-images.py \
  --page-id {作成されたセッション記録ページのID} \
  --after-heading "セッション画像" \
  --images "{セッションパス}/受講まとめ画像/{YYYY-MM-DD}_1.png" \
           "{セッションパス}/受講まとめ画像/{YYYY-MM-DD}_2.png" \
           ...（全画像ファイルを指定）
```

**スクリプトの処理フロー**:
1. `POST /v1/file_uploads` で File Upload オブジェクトを作成
2. `POST /v1/file_uploads/{id}/send` でファイルを送信
3. ページのブロックを取得し、`--after-heading` で指定した見出しブロックを検索
4. `PATCH /v1/blocks/{page_id}/children` で見出し直後に画像ブロックを挿入（`after` パラメータ使用）

**エラー時のフォールバック**:
スクリプトが失敗した場合（APIキー未設定、requests未インストール等）：
> 画像の自動アップロードに失敗しました。以下の画像をNotionページに手動でドラッグ&ドロップしてください：
> - {各画像ファイルのパス}
